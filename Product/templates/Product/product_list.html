{% comment %} action= IS LOAD-BEARING, DO NOT REMOVE IT. This form now sits inside the boosted #product-results region. Vendored htmx 2.0.3 boosts a form like this: o = getAttribute(form, "action"); if (verb === "get" && o.includes("?")) { ... } With no action attribute o is null, so o.includes() throws a TypeError — inside htmx's DOM-processing pass, which aborts. Every element after this form (the whole table and its pagination) then never gets wired up and falls back to native navigation: a full page reload on every filter. Silently — the only trace is one console error. htmx 2.0.4+ guards the null (path = location.href). Until we upgrade, every boosted form in this app MUST carry an explicit action. {% endcomment %}

{% comment %} hx-disinherit="*" IS THE LOAD-BEARING ONE — DO NOT DROP IT. "Archive" and "Create product" open MODALS via their own hx-get → #confirmBody. hx-boost="false" alone is NOT enough and shipping it that way BROKE them (07-17, user-caught): hx-select/hx-target/hx-swap/hx-push-url are INHERITED, and htmx resolves them by walking UP the tree (getClosestAttributeValue) until it finds a value. So these buttons silently picked up hx-select="#product-results" from the region, tried to lift #product-results out of the MODAL's response, found none, and swapped NOTHING into #confirmBody — main.html's htmx:afterSwap opened the modal anyway, so you got an empty panel over a dimmed page. hx-boost="false" only turns off BOOSTING; it says nothing about inheritance. hx-disinherit="*" makes the walk stop here, so these keep their own hx-get / hx-target / hx-swap (an element's OWN attributes are always found first) and inherit nothing from above — including hx-push-url, which would otherwise have pushed the modal's URL into the address bar. "Presets" is a plain that LEAVES the page; with inheritance cut it just navigates normally. {% endcomment %}
